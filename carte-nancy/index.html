Compte rendu du Projet

Ce projet s'inscrit dans la SAÉ d'Application Répartie. Il permet de visualiser sur une carte interactive Leaflet des données hétérogènes concernant la ville de Nancy, issues à la fois de données ouvertes publiques et d'une base de données privée via une architecture distribuée (RMI & Proxy HTTP).

Dépôt Git : https://github.com/JassemMT/S4-IL-02-Projet_application_repartie

1. Architecture de l'application

L'application est composée d'une interface web cliente (TypeScript/Leaflet) qui interroge d'une part les API publiques en direct, et d'autre part un serveur Proxy Java (gérant le CORS). Ce Proxy fait le pont avec un service RMI, qui lui-même communique avec une base de données Oracle.

Schéma de l'architecture distribuée

Frontend (TS/Leaflet) ──HTTP──► Proxy HTTP (Java) ──RMI──► Serveur RMI (Java) ──JDBC──► Oracle DB port 8080 port 1099
  • Frontend : carte interactive Leaflet en TypeScript, compilée avec esbuild
  • Proxy HTTP : passerelle Java (HttpServer) résolvant le CORS entre le navigateur et les services RMI
  • Serveur RMI : service Java exposant les méthodes getRestaurants() et reserverTable(...) via RMI
  • Base de données : Oracle, avec les tables RESTAURANT, TABLE_RESTAURANT, CRENEAU_TABLE, RESERVATION, PLAT, COMMANDE, CONTIENT

2. Ce qui a été réalisé

  • Base de données (Oracle) : Création d'un schéma relationnel avec contraintes (Tables RESTAURANT, TABLE_RESTAURANT, CRENEAU_TABLE, RESERVATION, PLAT, COMMANDE, CONTIENT). Un jeu de données a été inséré avec les coordonnées GPS réelles sur Nancy.
  • Service Backend RMI : Développement d'un service Java (ServiceRestaurantImpl) utilisant JDBC pour exposer deux méthodes distantes : la récupération des restaurants (format JSON), et la réservation d'une table. Les conflits concurrents sont gérés avec un FOR UPDATE ... WAIT sur la BDD.
  • Serveur Proxy HTTP : Création d'une passerelle (avec HttpServer) permettant de résoudre les problèmes de sécurité du navigateur (CORS). Il expose les services RMI sous forme d'API REST accessibles par le frontend.
  • Frontend (TypeScript / Leaflet) : Une carte interactive interrogeant de façon asynchrone (Fetch) l'API des VélOstan'lib pour afficher l'état des stations en temps réel avec des icônes SVG personnalisées.
  • Client de test : Un client RMI indépendant a été codé pour valider le bon fonctionnement métier sans passer par l'interface web.

6. Déploiement et Exécution

Pour lancer l'application complètement, plusieurs terminaux sont nécessaires :

Étape A : Préparer la base de données

Exécuter les scripts SQL dans l'ordre sur instance Oracle :

@database/script.sql
@database/insert.sql

Étape B : Compilation du backend Java (Maven)

À la racine du dossier RMI :

cd RMI
mvn clean package

Cela compile les trois modules (service, proxy, client) et produit les JARs dans leurs répertoires target/ respectifs.

Étape C : Lancement du Serveur RMI

Le serveur RMI doit être lancé en premier car le proxy s'y connecte au démarrage. Depuis le dossier RMI :

# Sur Windows :
java -cp "service/target/rmi-service-1.0.jar;service/ojdbc17.jar" sae.s4.rmi.service.LancerServeur

# Sur Linux / Mac :
java -cp "service/target/rmi-service-1.0.jar:service/ojdbc17.jar" sae.s4.rmi.service.LancerServeur

Il est possible de surcharger les identifiants de la base de données en ligne de commande :

java -cp "..." sae.s4.rmi.service.LancerServeur <user> <password>

Configuration du serveur RMI

Fichier RMI/service/src/main/resources/config.properties :

Propriété Description Valeur par défaut
db.url URL JDBC de la base Oracle (à adapter)
db.user Utilisateur Oracle (à adapter)
db.password Mot de passe Oracle (à adapter)
rmi.hostname Hostname RMI 127.0.0.1
rmi.port Port du registre RMI 1099
rmi.name Nom du service dans le registre serviceRestaurant

Étape D : Lancement du Proxy HTTP

Dans un autre terminal, depuis le dossier RMI :

java -jar proxy/target/rmi-proxy-1.0.jar

Configuration du proxy

Fichier RMI/proxy/src/main/resources/config.properties :

Propriété Description Valeur par défaut
rmi.host Hôte du serveur RMI localhost
rmi.port Port du registre RMI 1099
rmi.name Nom du service RMI serviceRestaurant
http.port Port d'écoute HTTP du proxy 8080
incidents.url URL de l'API Grand Nancy (incidents) (URL open data)
iut.proxy.host Proxy réseau IUT (vide si hors IUT) (vide)
iut.proxy.port Port du proxy IUT (vide si hors IUT) (vide)

Endpoints exposés par le proxy

Méthode Route Description
GET /restaurants Liste tous les restaurants (JSON)
POST /reserver Réserve une table (body JSON)
GET /incidents Incidents de circulation Grand Nancy

Exemple de body pour /reserver :

{ "nom": "Dupont", "prenom": "Jean", "convives": 3, "telephone": "0612345678", "idRestaurant": 1, "date": "2026-06-10", "heure": "19:00" }

Étape E : Test avec le Client RMI (Facultatif)

Pour vérifier le service RMI via le terminal :

java -jar client/target/rmi-client-1.0.jar

Il exécute automatiquement plusieurs scénarios de test (récupération des restaurants, réservation valide, restaurant inexistant, champs manquants, convives invalides).

Étape F : Compilation et Lancement du Frontend (Facultatif)

Dans le dossier carte-nancy, recompiler le TypeScript (si des modifications sont faites) :

cd carte-nancy
npm install
npm run build

Puis déposer le contenu du dossier carte-nancy/ sur Webetu (via SFTP ou le gestionnaire de fichiers de l'IUT).

URL de production : https://webetu.iutnc.univ-lorraine.fr/www/e52526u/nancy-carte/
